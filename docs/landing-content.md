## The Problem

### Section headline
AI builds UI. Just not *your* UI.

### Body
Ask any AI to build a form, a dashboard, or a nav — and it will. It'll also invent a color palette, make up spacing values, ignore your border radius scale, and produce components that look like no design system you've ever shipped.

The problem isn't the model. It's the missing context.

AI assistants don't know your tokens. They don't know your component rules. They don't know that your primary button is `--primary-600`, that your card border radius is `--border-radius-200`, or that secondary actions should never compete with the primary CTA on the same surface.

Without that context, every generation is a guess. You spend more time correcting output than using it.

### The fix
BrandSync gives the model the context it needs — directly, at inference time — so the output is consistent before you ever see it.

## What You Get

### Section headline
Everything the model needs to build on-brand UI.

### Feature 1 — Framework-native output

**Headline:** Code that goes straight into your repo.

**Body:** Claude generates components in the exact framework and syntax your codebase uses. React, HTML, or whatever your stack requires. No adapters, no translation layer, no reformatting. The output is shaped to fit where it lands.

**How:** `brandsync-mcp` exposes your component specs with anatomy, usage rules, and token references. The model reads the spec first, then generates the implementation — not the other way around.

---

### Feature 2 — Production-ready code

**Headline:** Every token resolved. Every rule respected.

**Body:** Colors reference `var(--primary-600)`. Spacing comes from `--spacing-300`. Border radii are `--border-radius-200`. The model can only reference tokens that exist in your system — structurally preventing drift.

**How:** `get_tokens` streams the live `brandsync-tokens` package to the model. No stale snapshots. No assumed values. Whatever is in your token file is what Claude works from.

---

### Feature 3 — Speed of development

**Headline:** From description to component in one prompt.

**Body:** Describe the screen, the state, or the interaction. Claude returns a complete, spec-accurate component — with the right layout, the right tokens, and the right component rules already applied. What used to take an hour of scaffolding takes seconds.

**How:** `get_component` returns the full spec for any Brandsync component in a single call — anatomy, states, variants, guidelines, accessibility notes, and token references. The model has everything it needs before it writes the first line.

---

### Feature 4 — Design system enforcement

**Headline:** Drift is structurally prevented.

**Body:** The model doesn't have to try to follow your design system. It can only use what your system defines. No invented tokens. No guessed values. No "close enough" colors. Consistency isn't a prompt instruction — it's a constraint built into the tool.

**How:** `search_guidelines` gives Claude access to your color, typography, and component usage rules as part of its context. The spec is always present, always accurate, and always enforced.

---

### Feature 5 — Live Figma sync

**Headline:** What's in Figma is what Claude builds.

**Body:** `figma-console-mcp` connects Claude directly to your Figma file. It can read your variables, extract component data, inspect current layouts, and — when you need it — create and edit designs directly inside Figma. The design file and the code stay in sync because they're reading from the same source.

**How:** Running locally via NPX or local git, `figma-console-mcp` gives Claude 94+ tools covering design extraction, variable management, component creation, accessibility scanning, and real-time monitoring. It bridges the gap between what the designer built and what the developer ships.
